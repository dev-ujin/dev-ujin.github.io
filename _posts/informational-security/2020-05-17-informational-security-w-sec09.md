# [1] 인증 개념
_____
## 1. 메시지 인증
- 전달되는 메시지의 이상유무(내용 변경, 순서 변경, 삭제 여부)를 확인할 수 있는 기능
- 메시지 암호화 방식, MAC(Message Authentication Code)방식, 해시 함수를 이용하는 방식 등이 있음

## 2. 사용자 인증
- 정당한 가입자의 접속인가를 확인하기 위한 인증
- 개인식별과의 차이점

# [2] 지식 기반 인증(Type 1)
_____
### 개념
- 사용자가 알고 있는 어떤 것에 의존하는 인증 기법(Something you know)
- 보안성은 비밀번호의 크기와 랜덤성에 달려 있음

### 장점
- 다양한 분야에서 사용 가능함 : 지식 기반 기법에 대한 신뢰도와 인식이 높음  
- 검증 확실성 : 검증 시 애매 모호성이 없음
- 관리비용의 저렴함 : 사용자의 기억과 시스템에서 보관하고 있는 참조 정보를 이용하므로 별도의 요금이 추가되지 않음

### 단점
- 소유자가 패스워드를 망각할 수 있음
- 공격자에 의한 추측이 가능함
- 사회 공학적 공격에 취약함

## 1. 패스워드(Password)
### 종류
1. 고정된 패스워드 : 접속 시에 반복해서 사용되는 패스워드
  - 패스워드 저장 형태
    1. 패스워드 평문 그대로
    2. 패스워드의 해시값
    3. (패스워드 + 솔트)의 해시값
      - UNIX에서 변형된 형태를 사용
    
2. 일회용 패스워드(OTP, One-Time Password) : 오직 한 번만 유효한 패스워드
  - 재전송 공격 방어
  - 패스워드 생성 방법
    1. 패스워드 목록 작성
    2. 패스워드를 순차적으로 업데이트(P<sub>i</sub>를 이용해 P<sub>i+1</sub>을 생성)
    3. 패스워드에 순차적으로 해시함수를 적용(h<sup>n</sup>(P<sub>0</sub>))
      - Leslie Lamport가 개발
  - 단말장치로 주로 구현하므로 소유기반(Type 2)으로 분류함 

### 문제점
- 추측이 쉬움
- NTCrack & John the ripper와 같은 소프트웨어로 크랙하기 쉬움
- 무작위로 만든 패스워드의 경우 기억하기 어려움

### 보안 정책
- 4가지 유형의 문자(대문자, 소문자, 숫자, 특수문자)로 구성
- 최소 8자리 이상의 문자
- 동일한 패스워드를 재사용하면 안되며, 공유되어서도 안됨
- 로그인 시도 횟수를 제한
- 단기간의 휴면 사용자 계정을 사용 불가능하게 하며, 더 이상 사용되지 않는 계정은 삭제

### 안전성
패스워드 추측 확률 P = (L &#215; R)/S
- L : 패스워드의 사용 기간
- R : 패스워드의 사용 빈도
- S : 패스워드의 길이

## 2. 시도-응답 개인 식별 프로토콜
- 대칭형 암호와 공개키 암호를 기반으로 함

### 종류
1. 일방향 개인 식별 프로토콜 : 서버나 클라이언트 중 한 대상을 식별하는 프로토콜
2. 상호 개인 식별 프로토콜
  - 서버가 클라이언트의 신원을 확인하는 것과 같이 클라이언트가 서버의 신원 확인을 원하는 경우 사용됨

## 3. 영지식 개인 식별 프로토콜(Zero-Knowledge Identification Protocol)
- 자신의 비밀 정보를 서버에게 제공하지 않고 신분을 증명하는 방식
- 클라이언트를 증명자(prover), 서버를 검증자(verifier)라고 함
- 스마트카드 및 원격지 로그인에서 사용
- 정당성(soundness), 완전성(completeness), 영지식성(zero-knowledgeness)을 만족해야 함

## 4. i-PIN(Internet Personal Identification Number)
- 인터넷상에서 주민번호 대신에 아이디와 패스워드를 이용해 본인 확인을 하는 수단

### 이용 효과
- 주민등록번호 유출예방
- 본인확인 강화






# [3] 소유 기반 인증(Type 2)
_____
### 개념
- 소유에 기반을 둔 인증 방식(Something you have)
- 도용의 위험이 있어 주로 Something you know나 Something you are 방식과 함께 쓰임

### 장점
- 일반적임 : 다양한 수단
- 입증된 기술
- 비용 측면에서 생체 인식방식보다 더 경제적

### 단점
- 소유물이 없을 경우 인증이 어려움
- 복제가 가능함
- 자산 관리 기능이 요구됨

## 1. 메모리 카드(토큰)
- 정보를 저장할 수 있지만 정보를 처리할 수는 없음
- 예 : 마그네틱선이 있는 은행 카드

## 2. 스마트카드
- 정보를 처리할 수 있다는 점에서 메모리 카드보다 발전된 기술
- 집적 회로가 내장된 신용카드 크기의 플라스틱 카드로서 접촉 혹은 비접촉 카드가 있음
- 장비보호 기능이 있어야 함

## 3. 일회용 패스워드(OTP, One-Time Password)
- 단말장치로 주로 구현되므로 소유기반으로 분류함

### 특징
- 일정시간마다 비밀번호를 변경
- 휴대폰을 통한 인증으로 사용자의 편리성 및 안전성을 확보

### 생성 및 인증 방식
1. 질의응답 방식
  - 서버가 사용자에게 시도(challenge, nonce)를 보내고 토큰장치는 사용자에게 임의값을 받아 OTP를 생성해서 보내줌
  - 사용자는 OTP와 사용자 이름을 인증 서버로 전송하고 인증서버는 이를 이전의 보낸 값과 비교하여 사용자를 인증함
2. 시간과 이벤트 동기화 방식
  - 시간 혹은 계수기를 이용한 인증서비스와 동기화가 핵심
  - 동일한 비밀키를 공유하여 암호화, 복호화에 사용
  - 질의응답 방식에 비해 호환성이 높음
3. S/KEY 방식
   - 클라이언트에서 정한 비밀키를 서버로 전송하고, 서버는 해시 체인 방식으로 비밀키에 대해 해시값을 구하는 작업을 n번 반복하고, n개의 OTP를 서버에 저장 
   - 유닉스 계열 운영 체제에서 사용되고 있음

### 단점
- 자신의 식별정보(ID, 사용자 이름)를 공유하거나 토큰장치를 공유 혹은 분실했을 경우 악용의 소지가 있음
- 토큰장치의 배터리 방전 혹은 오류로 인해 성공적인 인증이 어려울 수 있음




# [4] 개체(생물학적) 특성 기반 인증(Type 3)
_____
### 개념
- 개체의 생체적 특성에 기반을 둔 인증방식(Something you are)
- 지문, 홍채, 장문, 성문, 얼굴이미지 등
- 생체 템플릿(생체정보)을 데이터베이스에 등록한 후 사용
- 검증(Verification)과 신분확인(Identification)을 위해 사용됨

### 장점
- 사용하기 쉬움
- 잊어버리거나 도난당할 수 없음
- 위조가 어려움

### 단점
- 잘못 판단할 영역(판단 모호성)이 존재
- 관리가 어려움 : 생체 정보를 미리 저장소에 등록해야하고, 저장소는 생체 템플릿을 비밀스럽게 보관해야 함
- 인증을 위한 임계치 설정이 어려움

## 1. 생체인증(Biometrics)

### 개념
- 생체적 혹은 행동적 특성을 측정하여 신원을 인증하는 방법
- 추측하거나 훔치거나 공유할 수 없는 특성을 바탕으로 해 더 높은 보호를 제공함

### 평가 항목
- 보편성(Universality) : 모든 사람이 가지고 있는 생체 특징인가?
- 유일성(Uniqueness) : 동일한 생체 특징을 가진 타인은 없는가?
- 지속성(영구성, Permanenece) : 시간에 따른 변화가 없는 생체 특징인가?
- 획득성(Collectability) : 정량적으로 측정이 가능한 특성인가?
- 성능(Performance) : 환경변화와 무관하게 높은 정화성을 얻을 수 있는가?
- 수용성(Acceptability) : 사용자의 거부감은 없는가?
- 반기만성(Anti-Circumvention) : 고의적인 부정사용으로부터 안전한가?

### 정확도
- 오거부율(부정거부율, False Rejection Rate) : 인식되어야 할 사람이 얼마나 자주 인식 되지 않는지를 나타내는 값
- 오인식률(부정허용율, False Acceptance Rate) : 인식되어서는 안 될 사람이 얼마나 자주 인식이 되는지를 나타내는 값
- 시스템에 민감할수록 오거부율이 높아지고, 시스템의 민감도가 떨어지면 오인식율이 높아짐





# [5]] 통합 인증 체계(SSO, Single Sign On)
_____
### 개념

### 구성요소
1. 사용자
2. 인증 서버
3. LDAP(Lightweight Directory Access Protocol) : X.500을 근거로 한 디렉터리 데이터베이스에 접속하기 위한 통신 규약
4. SSO Agent : 각 정보시스템에 자동인증 정보(Token) 송수신 수행

### 장점
- 운영비용 감소
- 보완성 강화
- 사용자 편의성 증가(패스워드 분실 감소)
- 중앙집중 관리를 통한 효율적 관리 기능

### 단점
- SSO 서버가 단일실패지점 : ID, PASSWORD 노출 시 전체시스템 위험
- SSO 서버 침해 시 모든 서버의 보안 침해 가능
- SSO 개발 및 운영비용 발생
- 자원별 권한관리 미비

## 1. 커버로스(Kerberos)
### 개념
- 클라이언트/서버 모델에서 동작하며 대칭키 암호기법에 바탕을 둔 티켓 기반 인증 프로토콜이며 동시에 KDC(Kerberos 버전 4는 DES를 사용)

### 특징
- 개방형 아키텍처 : 공급업체들이 그들의 제품과 환경 내에서 올바르게 동작시키기 위해 조작할 수 있음
- 기업 접근 통제를 위해 확장성, 투명성, 안정성, 보안의 요소를 가짐
- 유효기간 내의 티켓을 사용하여 동일한 서버에서 여러 가지의 응용 서비스를 받을 수 있음
- 사용하는 이유 : 주체들이 직접 통신하기에 서로를 충분히 신뢰하지 않음. 이 경우에 주체들은 오직 KDC만을 신뢰함

### 구성요소
1. KDC(Key Distribution Center)
   - 키 분배 서버(가장 중요한 요소)
   - 모든 사용자와 서비스의 비밀키를 보관
   - 티켓을 생성하고 인증서비스를 제공
2. AS(Authentication Service)
   - 실질적으로 인증을 수행
   - 사용자에 대한 인증을 수행하는 KDC는 부분 서비스
3. TGS(Ticket Granting Service)
   - 티켓 부여 서비스
   - 티켓을 부여하고 분배하는 KDC의 부분 서비스 
4. Ticket
   - 사용자에 대해 신원과 인증을 확인하는 토큰
   - 사용자가 다른 주체들과 통신이 필요할 때마다 패스워드를 입력하지 않도록 도와 줌
5. Principals
   - 인증을 위하여 커버로스 프로토콜을 사용하는 모든 실체


### 구성도
- AS는 각 서버와 유일한 비밀키를 공유
- 사용자는 우선 AS에게 티켓 승인 티켓(Ticket-Granting Ticket)을 요청하고, 사용자 워크스테이션의 클라이언트 모듈이 이 티켓을 보관함
- 사용자는 자신을 인증하는 티켓을 이용하여 TGS에 접속하고, TGS는 요청된 서비스에 대한 티켓을 발행함
- 클라이언트는 각 서비스-승인 티켓(Service-Granting Ticket)을 보관함
- 타임스탬프를 이용한 시간제한을 두어 위장을 막음 

### 취약점
- 패스워드 추측 공격에 취약
- 사전공격이 발생하고 있어도 알지 못함

### 장점
- 데이터의 기밀성과 무결성 보장
- 재생공격 예방
- 개방된 이기종 간의 컴퓨터에서 자유로운 서비스 인증이 가능(SSO)
- 대칭키를 사용하여 도청으로부터 보호

### 단점
- 비밀키, 세션키가 임시로 단말기에 저장되어 침입자에 의해 탈취당할 수 있음
- 타임스탬프로 인해 시간동기화 프로토콜이 필요
- KDC는 많은 수의 요청을 처리 가능해야 함(확장 가능성이 있어야 함)
- TGS & AS는 물리적 공격 및 악성코드로부터의 공격에 취약

## 2. 세사미(SESAME, Secure European System for Application in a Multi-vendor Environment)

### 개념
- 커버로스의 기능을 확장하고 약점을 보완하기 위한 SSO기술
- 대칭 및 비대칭 암호화 기법을 사용하여 인증서비스를 제공

### 커버로스와 차이점
- 커버로스는 주체를 객체에 대하여 인증시키기 위해 티켓을 사용하는 반면에, 세사미는 PAC(Privileged Attribute Certificate)를 사용하며, PAC는 주체의 신원, 객체에 대한 접근 능력, 접근 기간 그리고 PAC의 유효 기간을 포함
