#### 1. 키 배송 문제
- 대칭키 암호를 사용하려고 하면 `키 배송 문제`가 발생

##### 1. `키의 사전 공유에 의한 해결`
- `키 관리기관(TA, Trusted Authority)`이 사전에 두 사용자 A, B에게 비밀 경로를 통하여 임의 키 K<sub>A, B</sub> = K<sub>B, A</sub>를 선택하여 전달하는 방법
- 사용자 사이에 안전한 통로가 필요
- **TA는 물론 사용자들도 많은 키를 관리해야하는 문제점이 발생**

#####  2. `키 배포 센터(KDC, Key Distribution Center)에 의한 해결(온라인 키 분배)`
- **신뢰받는 제 3자**에 의뢰해서 개인과 키배포 센터 사이에서만 키를 사전에 공유하는 것
- <r>네트워크의 모든 사용자와 필요할 때마다 키를 공유하는 방법</r>
###### 절차
1. 앨리스가 KDC에 '밥과 통신하고 싶다'고 요청
2. KDC는 의사난수 생성기를 사용하여 앨리스와 밥의 이번 통신만을 위한 일시적인 키인 세션키(K)를 생성
3. KDC는 데이터베이스에서 앨리스의 키와(KA)와 밥의 키(KB)를 꺼냄
4. KDC는 앨리스의 키를 사용하여 세션키를 암호화(C<sub>A</sub> = E<sub>KA</sub>(K))해서 앨리스에게 보냄
5. KDC는 밥의 키를 사용하여 세션키를 암호화(C<sub>B</sub> = E<sub>KB</sub>(K))해서 밥에게 보냄
6. 앨리스는 KDC에서 받은 세션키를 복호화(K = D<sub>KA</sub>(C<sub>A</sub>))해서 세션키를 얻음
7. 앨리스는 세션키를 사용하여 밥에게 보낼 메일을 암호화(C = E<sub>K</sub>(M))함
8. 밥은 KDC에서 받은 세션키를 복호화(K = D<sub>KB</sub>(C<sub>B</sub>))해서 세션키를 얻음
9. 밥은 앨리스에게 받은 암호문을 세션키를 사용하여 복호화(M = D<sub>K</sub>(C))함
10. 앨리스와 밥은 세션키를 삭제함

##### 3. `Diffie-Hellman 키 교환에 의한 해결`

###### Diffie-Hellman 개념
- **공개키 암호방식의 개념을 사용**
- **KDC없이 대칭 세션키를 생성**
  - <r>세션키는 암호화하여 전달할 필요가 없음</r>
  - 대칭키를 만들기 전에 양쪽은 두 개의 수 p와 g를 선택하는데, p는 매우 큰 소수로서 300자리가 넘는 십진수(1024비트)
    - <r>p와 g는 사전에 공유하여야 함</r>
    - <bl>p와 g는 비밀로 할 필요가 없어 도청자 이브에게 알려져도 상관 없음</bl>
- **이산대수 문제를 풀기 어렵다는 사실에 근거하고 있음**
  - 이산대수 문제 : G<sup>A</sup> mod P에서 수 A를 구하는 문제
- <r>정보의 기밀성을 제공하지 못하기 때문에 데이터에 암호화나 복호화에는 사용할 수 없음</r>
- <r>신분 위장이나 재전송 공격에 약함</r>
- <r>사용자 A와 B가 각각 랜덤 수 x, y를 선택한 경우, B는 공개된 정보로부터 g<sup>xy</sup> mod p를 쉽게 얻을 수 있음</r>
###### 절차
1. 앨리스는 임의의 큰 수 x를 0 &#8804; x &#8804; p-1안에서 선택하고 R<sub>1</sub> = g<sup>x</sup> mod p를 계산
2. 밥은 다른 임의의 큰 수 y를 0 &#8804; y &#8804; p-1안에서 선택하고 R<sub>2</sub> = g<sup>y</sup> mod p를 계산
3. 앨리스는 R<sub>1</sub>를 밥에게 보냄
4. 밥은 R<sub>2</sub>를 앨리스에게 보냄
5. 앨리스는 K = (R<sub>2</sub>)<sup>x</sup> mod p를 계산
6. 밥은 K = (R<sub>1</sub>)<sup>y</sup> mod p를 계산
7. 이렇게 구한 K는 아래와 같고 이것이 세션에 사용될 대칭키가 됨<br>
K = (g<sup>x</sup> mod p)<sup>y</sup> mod p = (g<sup>y</sup> mod p)<sup>x</sup> mod p = g<sup>xy</sup> mod p 

###### 안전성
- 이산대수 공격이 발생할 수 있음
  - 이브가 R<sub>1</sub>과 R<sub>2</sub>를 가로챌 경우
- `서비스 거부 공격(DoS, Denial of Service)`가 발생할 수 있음
  - DH 기법은 지수 함수에 기초하고 있으므로 제3자가 공격 대상 서버에 대해 IP 스푸핑을 통해 위조한 키 생성 요청을 동시에 다수 요청함으로써 **키 생성 부담으로 서버가 마비되도록 공격**할 수 있음
  - 방지하기 위해서는 키 생성 요청자를 확인하기 위한 `쿠키(Cookie)`를 사용하면 됨
- `중간자 공격(man-in-the-middle attack)`
  - **인증단계가 없기 때문에 이런 공격에 취약함**
  - 방지하기 위해서는 `전자서명`과 `공개키 인증서` 등을 이용하면 됨 

##### 4. `공개키 암호에 의한 해결`
- 송신자는 수신자로부터 사전에 공유 받은 암호화키로 메시지를 암호화하여 수신자에게 보냄
- 암호화키는 도청자에게 알려져도 상관 없음
- 복호화는 복호화키를 가지고 있는 수신자만이 가능
- 이렇게 하면 복호화키를 수신자에게 배송할 필요가 없어짐

#### 2. 비대칭키 암호(공개키 암호)
##### 비대칭키 암호의 개념
- **수학적으로 해결하기 곤란한 문제를 토대로 해서 기밀성을 유지**

###### 비대칭키 암호에서 키
- **암호화키와 복호화키가 분리되어 있음**
- 키쌍을 이루고 있는 2개의 키는 서로 밀접한 수학적 관계가 있어서 공개키와 개인키를 각각 별개로 만들 수는 없음

###### 비대칭키 암호의 등장으로 인한 오해
- 비대칭키(공개키) 암호시스템의 등장으로 인해 대칭키(비밀키) 암호시스템의 역할이 없어질 것이라는 오해가 있지만 사실은 그렇지 않음
- 비대칭키 암호시스템은 수학적인 함수를 사용하므로 **처리속도가 느려서** 메시지의 크기가 큰 경우에는 대칭키 암호시스템의 역할이 매우 중요하기 때문

##### 1. `RSA 암호`
###### RSA 암호의 개념
- **세계적으로 사실상의 표준**
- **대표적인 공개키 암호 알고리즘의 암호화뿐만 아니라 전자서명의 용도로도 사용됨**
- **인수분해 문제가 어렵다는 사실에 근거**
- 응용 : SSL 프로토콜을 가진 많은 웹 브라우저, PGP, 공개키 암호시스템을 사용하는 정부시스템

###### RSA 암호의 암호화와 복호화
- 두 개의 지수 e와 d를 사용하는데, e는 공개하는 값이고 d는 비밀을 유지하는 값
- 앨리스는 C = P<sup>e</sup> mod n을 이용하여 암호문 C를 생성하고, 밥은 P = C<sup>d</sup> mod n을 구하여 평문 P를 얻음
- 모듈러 n은 매우 큰 수이고 키 생성 프로세스를 통해서 만들어짐

###### RSA 암호의 알고리즘 예
1. 두 소수 p = 17과 q = 11을 선택
2. N = p * q = 17 * 11 = 187을 계산
3. &#8709;(N) = (p-1)(q-1) = 16 * 10 = 160을 계산
4. &#8709;(N) = 160보다 작으면서 &#8709;(N)과 서로소인 수 e를 선택. 여기서는 e = 7을 선택
5. d <ㄱ 160이면서 de mod 160 = 1인 수 d를 결정. 23 * 7 = 161 = 1 * 160 + 1이기 때문에 d = 23
6. 결과로 얻어지는 공개키 **PU = {e, n}** = {7, 187}이고, 개인키는 **PR = {d, n}** = {23. 187}

- RSA 알고리즘에서 ***공개되는 것은 n, e**이고 **비밀이 유지되는 것은 p, q, d, &#8709;(n)**
- **암호화 : C = P<sup>e</sup> mod n**
- **복호화 : P = C<sup>d</sup> mod n**

###### RSA에 대한 공격
1. `수학적 공격(소인수분해 공격)` : 두 개의 소수 곱을 인수분해 하려는 공격
   - 방지하기 위해서는 **키의 길이가 긴 것을 사용해야 함**
   - 키 길이가 길어지면 암호화, 복호화에서 계산량 또한 커지기 때문에 **처리 속도는 느려짐** 
   - d의 비트 수가 크면 클수록 더 안전함
2. `타이밍 공격(시간 공격)`
   - 복호화 알고리즘의 실행 시간에 따라 달라짐
   - 소요되는 시간이 드러나지 않게 막아 키 길이를 추측하는 공격을 막을 수 있음
3. `선택 암호문 공격(CCA)` : 임의의 데이터를 송신하면 그것을 암호문으로 간주하고 회신해 주는 서비스
   - 공격자는 위조 암호문을 여러 차례 서버에 보내 서버가 회신해 주는 **오류 메시지나 타이밍을 해석**해서 K나 P의 정보를 조금이라도 얻을 수 있음
   - 방지 위해서는 `최적 비대칭 암호화 패딩(OAEP, Optimal Asymmetric Encryption Padding)`을 이용을 권장

###### `최적 비대칭키 암호화 패딩(OAEP, Optimal Asymmetric Encryption Padding)`
- **RSA를 개량한 알고리즘**
- 복호화에서 평문 해시값과 정해진 개수의 '0'으로 만들어진 인증정보를 평문의 앞에 추가하고 RSA로 암호화함
- 오류 메시지를 만들 때는 **어떤 오류인지 구체적으로 알리지 않는 것이 중요**
- **난수를 이용**해서 암호문이 매번 다른 패턴이 되도록 하여 안전성을 높임

##### 2. `Rabin 암호시스템(Rabin Cryptosystem)`

###### Rabin 암호시스템의 개념
- **소인수분해 문제가 풀기 어렵다는 것에 근거**
- RSA 암호 방식에서의 공개키 (n, e)의 e를 b로 바꾼 것과 같음

###### Rabin 암호시스템의 암호화와 복호화
- **연산은 오직 한 번의 곱셈으로 이루어져 있기 때문에 매우 간단하며, 매우 빠름**
- **성능이 낮은 플랫폼**에서 잘 활용될 수 있음
  - 매우 제한된 메모리를 가지고 있어 CPU 연산이 빨라야 하는 스마트카드에서 이용됨

###### Rabin 암호시스템의 보안
- p와 q가 충분히 크기만 하다면 안전함

##### 3. `ElGamal 방식`

###### ElGamal 방식의 개념
- **이산대수 문제에 근거해서 만든 시스템**
- **Diffie-Hellman 알고리즘의 확장**
- 단점 : 다른 알고리즘과 비교할 때 **가장 느림**
- 응용 : 디지털 서명, 암호화, 키 교환, GnuPG

##### 4. `타원 곡선 암호(ECC, Elliptic Curve Cryptosystem)`

###### 타원 곡선 암호의 개념
- **키의 길이가 짧아도** 되는 암호시스템
- **타원곡선 군에서의 이산대수 문제에 기초**
- **HW와 SW로 구현하기가 용이**
- <r>전자서명에 주로 사용됨</r>
- 응용 : 전자상거래

|항목|RSA 방식|ECC 방식|
|---|---|---|
|기반구조|**PKI(유선)**|**WPKI(무선)**|
|속도|느림|우수|
|키 크기|ECC에 비해 큼|상대적으로 작음|
|적용|**인프라가 다소 구현된 환경**|**소형 모바일 환경**|

##### 5. 기타 알고리즘
###### `Goldwasser-Micali`
- <r>**C로부터 P의 어떤 부분의 정보도 노출되지 않는 암호방식**</r>
- <r>**인수분해 문제와 제곱 잉여의 원리를 이용한 확률 암호를 정의**</r>

##### 공개키 암호알고리즘 분류
1. 소인수분해 문제 : RSA, Rabin, Goldwasser-Micali
2. 이산대수 문제 : ElGamal, DSA, ECC

#### 4. `하이브리드 암호시스템`
- 대칭키 암호를 사용하면 기밀성을 유지하며 통신할 수 있지만, 키 배송문제를 해결해야 함
- 키 배송문제를 해결하기 위해 공개키 암호를 사용할 수 있지만, 처리 속도가 매우 느림
- 이를 해결하기 위해 하이브리드 암호시스템을 사용

###### 하이브리드 암호시스템의 개념
- **대칭키 암호와 공개키 암호의 장점을 살린 암호시스템**
- 대칭키 암호로 메시지를 암호화를 하되, 대칭키 암호키는 공개키 암호로 암호화함
- 사용되는 기술
  1. **의사난수 생성기(대칭키 암호에 사용되는 세션키를 의사난수 생성기를 이용해 생성)**
  2. **대칭키 암호(메시지를 대칭키 암호방식으로 암호화)**
  3. **공개키 암호(대칭키 암호에 사용되는 세션키를 공개키 암호로 암호화)**

###### 하이브리드 암호시스템의 응용
- PGP, SSL/TLS
- PGP에서는 전자서명, 전자서명의 검증, 개인키의 관리도 추가하여 사용함
